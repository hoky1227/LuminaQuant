## RALPLAN-DR Summary

### Principles

1. **One portfolio optimization contract**: common split handling, weighting, constraints, memory evidence, and report schema should live in shared code instead of ad-hoc scripts.
2. **OOS lockbox by default**: optimizers may report OOS, but selection objectives should be train/validation unless explicitly labeled diagnostic.
3. **8GB is a hard runtime budget**: every heavy path must keep memory guard evidence and should avoid unbounded grids, materialized duplicate arrays, and full-cache commits.
4. **Artifacts are first-class**: every long run must leave restart-safe JSON/MD/log evidence under `var/reports/...` or `.omx/...`.
5. **AGENTS.md must route context**: include a bounded repo tree and ownership map so future sessions know where portfolio, research, data, live, tests, and artifacts live.

### Decision Drivers

1. Avoid recurring CI emails by preserving green CI and waiting for Actions after each pushed integration wave.
2. Reduce duplicated optimizer/tuner logic before adding more strategies.
3. Make the next agent’s first hour deterministic: tree, commands, state, and verification gates should be in tracked files.

### Viable Options

#### Option A — Conservative Shared-Core Extraction (Recommended)

- Extract shared optimizer utilities from `scripts/run_portfolio_optimization.py` and `scripts/research/tune_profit_moonshot_fresh_portfolio.py` into a small module, likely `src/lumina_quant/portfolio/optimizer_core.py` or `src/lumina_quant/research/portfolio_optimizer.py`.
- Keep script CLIs stable and adapt them to the shared module.
- Add tests around shared behavior before switching call sites.

Pros:
- Lowest risk to current green gate.
- Easier review and rollback.
- Directly targets duplication and future agent confusion.

Cons:
- Does not fully redesign all research scripts in one pass.
- May leave some legacy scripts as wrappers for a while.

#### Option B — Full Research Stack Rewrite

- Redesign replay, tuner, portfolio optimizer, Optuna, validator, and reporting around a new orchestration module.

Pros:
- Cleanest long-term architecture if executed perfectly.
- Could remove a lot of script-level glue.

Cons:
- High regression/CI risk.
- Hard to preserve all existing artifact contracts.
- More likely to violate OOS/8GB constraints during migration.

#### Option C — Rust/Native Hot Path First

- Port stream aggregation / combination / metric loops to Rust or native backend before module cleanup.

Pros:
- Potential maximum speedup on numeric hot loops.
- May reduce Python allocation overhead.

Cons:
- Premature if the main bottleneck is duplicated Python IO/schema work.
- Adds build/CI complexity and cross-platform risk.
- Should be deferred until profiling proves Python/vectorization is insufficient.

### Recommendation

Choose Option A first. Add profiling and benchmark evidence. Only escalate to Option C if shared-core/vectorized Python cannot meet measurable targets under 8GB.

## ADR

### Decision

Create a shared, memory-safe portfolio optimization core and convert existing script entrypoints into thin wrappers while preserving their CLI contracts and artifacts. Add an `AGENTS.md` repository tree/ownership section before broad execution so subsequent agents use current context instead of stale hidden state.

### Drivers

- The repo already has multiple portfolio optimizers/tuners with overlapping split/weight/metric/report logic.
- The latest moonshot pass succeeded partly because Optuna replaced broad grids and memory guards were enforced.
- Future sessions need deterministic context and guardrails to avoid repeating stale-context failures.

### Alternatives Considered

- Full rewrite: rejected for high regression risk before a complete code map and tests.
- Rust-first port: rejected until profiling identifies stable hot loops and CI/build surface is acceptable.
- Keep adding one-off scripts: rejected because it compounds context drift and duplicated OOS/memory policy.

### Why Chosen

Shared-core extraction maximizes correctness and maintainability while keeping existing successful artifacts and CI green. It also creates clean seams for later Rust/vectorized acceleration.

### Consequences

- Some scripts will remain as compatibility wrappers temporarily.
- Tests must verify both old CLI outputs and new shared-core semantics.
- AGENTS.md will become a maintained context surface; update it when major tree ownership changes.

### Follow-ups

- Add performance benchmarks for portfolio stream alignment/aggregation and combination.
- Consider a later native backend only for proven hotspots.
- Add a periodic “repo tree refresh” command/doc entry if the tree changes often.
